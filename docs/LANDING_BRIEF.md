## Tema (auto + toggle) e 3D

TEMA AUTO+TOGGLE SEM FOUC (infra ~1h, zero risco; o gerar-headers.mjs JÁ hasheia scripts is:inline sozinho — ZERO mudança de CSP):
1. Script anti-FOUC no <head> do Layout.astro ANTES do CSS, com is:inline (obrigatório — sem isso o Astro externaliza com defer e o flash volta): lê localStorage 'olli-tema'; se vazio cai em matchMedia('(prefers-color-scheme: dark)'); aplica data-tema no <html>. <1KB.
2. TRÊS ESTADOS (regra do projeto 'erro não vira vazio'): null=segue o sistema, 'claro', 'escuro'. Enquanto null, listener matchMedia('change') reage se o SO trocar. localStorage SEMPRE em try/catch — erro NUNCA vira 'claro' gravado, segue o sistema sem gravar (modo privado lança exceção).
3. Botão de troca no header com SVGs sol/lua trocados por CSS puro ([data-tema=escuro] .icone-sol{display:none}). Handler num <script> NORMAL do Astro (vira arquivo externo permitido por 'self') com addEventListener — PROIBIDO onclick= inline (hash de CSP não cobre atributo de evento; 'unsafe-hashes' é regressão de segurança).
4. CSS no global.css: @custom-variant dark (&:where([data-tema=escuro], [data-tema=escuro] *)); + bloco :root[data-tema=escuro]{ color-scheme:dark; --color-ink:#e8eef7; --color-paper:#0d1726; --color-line:#1e2d42; --color-tint:#122238; --color-muted:#8fa3bd; --color-slate:#a9bcd4 }. Como as utilities do Tailwind v4 compilam pra var(), isso re-tematiza ~70% sem tocar em classe. Depois, passada com dark: nos hardcoded do index.astro (nav bg-white/80, cards bg-white, border-white/70, sombras rgba). REMOVER o color-scheme:light fixo atual do global.css.
5. Layout.astro: trocar o meta theme-color único por DOIS com media (light #0b6fce / dark #0a2547) + update via JS no toggle manual.
6. ESCOPO ESPERTO: mockups do HeroDevices ficam INVARIANTES ao tema (objetos físicos aria-hidden, phone já é escuro — corta o trabalho de paleta pela metade). Custo honesto: infra ~1h; PALETA escura completa + QA de contraste = 0,5-1 dia (é o que realmente custa).

3D DOS MOCKUPS (upgrade no HeroDevices.tsx, SEM biblioteca nova — motion/react já está no bundle):
NÃO adotar three.js/R3F/Spline: >150KB gzip, game-loop 60fps fere a regra 'sem loop contínuo', Spline baixa .splinecode de CDN que o connect-src 'self' do CSP BLOQUEARIA, e o alvo Android intermediário paga em LCP/INP. CSS 3D transforms + motion cobrem 100% do efeito.
(a) PROFUNDIDADE REAL: translateZ distinto por camada dentro do pai preserve-3d (browser ~-40px, phone ~+60px, blobs ~-80px) = paralaxe verdadeira no tilt. ARMADILHA: overflow-hidden/rounded força flat e mata o preserve-3d dos filhos — aplicar translateZ em wrappers FORA dos elementos com overflow-hidden.
(b) GLARE DINÂMICO: ligar o gradiente de vidro do phone (hoje estático) aos MotionValues px/py via useTransform — maior percepção de 3D por custo quase zero.
(c) SOMBRA REATIVA sob o phone (camada própria animando só transform+opacity — NUNCA box-shadow) + ESPESSURA (frame duplicado translateZ(-6px) mais escuro).
(d) MOBILE (sem mouse): tilt ligado ao scroll com useScroll+useTransform (user-driven, não é loop — cumpre a regra). NÃO apostar em CSS animation-timeline:view() como via única (Firefox stable atrás de flag em jun/2026); se usar, @supports + fallback.
GUARDRAILS: animar SÓ transform/opacity; máx ~20 camadas compostas (translateZ em 3-5 wrappers, não por elemento); will-change só onde medido; reduzir/esconder os blobs blur-3xl gigantes em telas pequenas; gate matchMedia('(pointer: fine)') no parallax; manter useReducedMotion (pose 3D estática ok, movimento não); H1 NUNCA entra de opacity:0 (LCP deve ser o texto); validar com CPU 4x throttle + paint flashing.
